# PGP Gateway Implementation Plan

## Goal

Add a minimal SMTP sidecar that encrypts outbound mail before it reaches
SMTP2GO.

The gateway exists only to protect message content from the upstream SMTP relay.
Maddy remains the cluster-internal submission service and authentication boundary.
SMTP2GO remains the public outbound relay and DKIM signer.

Target flow:

```text
workloads -> Maddy mail-relay -> pgp-gateway sidecar -> SMTP2GO -> recipient
```

## Non-Goals

- Do not replace Maddy as the app-facing SMTP relay.
- Do not expose the PGP gateway as a Kubernetes Service.
- Do not add SMTP authentication on the Maddy-to-gateway hop.
- Do not support arbitrary unconfigured recipients or key discovery.
- Do not fetch public keys from keyservers.
- Do not store or mount private keys in the cluster.
- Do not sign OpenPGP messages in the gateway.
- Do not do DKIM signing in the gateway. SMTP2GO signs the final encrypted
  message.

## Design Principles

- Keep the Go code ordinary and boring: small packages, explicit errors,
  table-driven tests, standard formatting, no framework layer.
- Prefer maintained libraries for SMTP and OpenPGP. Do not hand-roll protocol
  parsing or cryptography.
- Keep the gateway fail-closed. If encryption or SMTP submission fails, the
  message must not be sent in plaintext.
- Keep logs free of message bodies, PGP material, and SMTP credentials.
- Make configuration obvious from environment variables and mounted files.
- Bind the inbound SMTP listener to `127.0.0.1` only.

## Libraries

Preferred starting point:

- `github.com/emersion/go-smtp` for the local SMTP server and outbound SMTP
  client.
- `github.com/emersion/go-sasl` for outbound SMTP AUTH PLAIN.
- `github.com/emersion/go-message` for MIME parsing and construction.
- `github.com/ProtonMail/gopenpgp/v3` for OpenPGP encryption.
- `github.com/prometheus/client_golang/prometheus/promhttp` for the metrics
  endpoint.

Avoid `golang.org/x/crypto/openpgp` for new code. It is deprecated and only
maintained for security fixes.

## SMTP Behavior

### Inbound From Maddy

The gateway listens on `127.0.0.1:2525` inside the `mail-relay` Pod.

Inbound behavior:

- No STARTTLS.
- No SMTP AUTH.
- No internal queue or spool file. The gateway is a synchronous filter.
- Return SMTP success to Maddy only after SMTP2GO accepts the encrypted message.
- Accept exactly one message transaction at a time per connection.
- Accept exactly one envelope recipient per message.
- Select the encryption key by matching the recipient against static key rules.
- Reject any recipient that does not match exactly one configured key rule.
- Reject empty senders unless a concrete app needs bounce-style mail.
- Reject messages larger than `MAX_MESSAGE_BYTES`.
- Return `5xx` for policy failures.
- Return `4xx` for temporary upstream submission failures.

This hop is localhost-only and exists inside the same Pod as Maddy. Maddy is
already responsible for authenticating workloads and authorizing sender domains.

### Outbound To SMTP2GO

SMTP2GO advertises:

```text
250-SIZE 52428800
250-8BITMIME
250-AUTH SCRAM-SHA-1 SCRAM-SHA-256 CRAM-MD5 PLAIN LOGIN
250-STARTTLS
```

Use the simplest supported secure path:

- Connect to `mail.smtp2go.com:587`.
- Require STARTTLS before authentication.
- Use SMTP AUTH PLAIN over the TLS connection.
- Use the original envelope sender from Maddy.
- Use the original envelope recipient unless the matched key rule explicitly
  rewrites it to a canonical delivery address.
- Do not implement SCRAM unless PLAIN stops working for SMTP2GO.

PLAIN is acceptable here because credentials are sent only after STARTTLS has
completed and certificate verification has succeeded.

## PGP/MIME Construction

The gateway emits PGP/MIME, not inline PGP.

Outer message:

- `From`: preserve the original message `From`.
- `To`: original recipient, unless the matched key rule sets `relayTo`.
- `Subject`: configured fixed value, default `Encrypted message`.
- `Date`: generated by the gateway.
- `Message-ID`: generated by the gateway.
- `MIME-Version: 1.0`.
- `Content-Type: multipart/encrypted; protocol="application/pgp-encrypted"`.

Encrypted payload:

- Preserve the original MIME body structure, including attachments, inside the
  encrypted OpenPGP payload.
- If the original message is not MIME, synthesize a `text/plain; charset=utf-8`
  inner part.
- Do not log or expose the decrypted inner content.

The outer SMTP headers remain visible to SMTP2GO because SMTP requires them for
transport and delivery. The outer subject is intentionally boring so application
subjects are not exposed to the upstream relay.

The PGP/MIME body structure should follow RFC 3156:

```text
multipart/encrypted; protocol="application/pgp-encrypted"
  application/pgp-encrypted
    Version: 1
  application/octet-stream; name="encrypted.asc"
    -----BEGIN PGP MESSAGE-----
```

## Key Rules

Keys are configured explicitly. A public key may be used for addresses other
than the identity printed on the key. This supports catchall mailboxes where
multiple recipient addresses arrive in the same inbox but only the main mailbox
address has a published OpenPGP key.

Use a small mounted YAML file:

```yaml
rules:
  - match:
      - main@example.com
      - "*@example.com"
    keyFile: /etc/pgp-gateway/keys/main.asc
    fingerprint: "0123456789ABCDEF0123456789ABCDEF01234567"
    relayTo: ""
```

Rule behavior:

- Normalize recipient addresses to lowercase before matching.
- Match exact addresses and `*` wildcards against the full address.
- Evaluate all rules and require exactly one match.
- `relayTo: ""` preserves the original envelope recipient.
- A non-empty `relayTo` rewrites only the outbound envelope and `To` header, not
  the encrypted inner message.
- Validate every configured public key and fingerprint at startup.

## Configuration

Environment variables and sane defaults:

```text
LISTEN_ADDR=127.0.0.1:2525                  # default
METRICS_ADDR=:9090                          # default
PGP_KEY_RULES_FILE=/etc/pgp-gateway/keys.yaml
OUTER_SUBJECT=Encrypted message             # default
MAX_MESSAGE_BYTES=26214400                  # default: 25 MiB
SMTP_HOST=mail.smtp2go.com                  # default
SMTP_PORT=587                               # default
SMTP_USERNAME=<from Secret>                 # required
SMTP_PASSWORD=<from Secret>                 # required
SMTP_TIMEOUT=30s                            # default
```

Public keys are not secret material. Store the key rules and public key material
in a Kubernetes Secret mounted into the sidecar. It is acceptable for that Secret
manifest to contain the public key value directly in the repo. Fingerprints
should still be configured and verified at startup so a bad key fails fast.

Only SMTP credentials are actually sensitive. Everything else should be
configurable through environment variables with defaults that work for the
current SMTP2GO/Maddy deployment.

SMTP2GO credentials continue to come from the existing `mail-relay` Secret
managed by External Secrets and SSM.

## Kubernetes Wiring

Update `services/platform/mail-relay`:

- Add the `pgp-gateway` container to the existing Maddy Deployment.
- Mount the key rules file and public keys from a Kubernetes Secret into the
  sidecar.
- Reuse the existing upstream SMTP credential Secret for the sidecar.
- Change Maddy's `deliver_to smtp` target from SMTP2GO/Proton directly to
  `tcp://127.0.0.1:2525`.
- Remove upstream SMTP auth from the Maddy outbound hop; the sidecar owns
  SMTP2GO authentication.
- Keep the existing cluster-internal Maddy Service name, selector, and SMTP port
  unchanged.
- Add a `metrics` port to the existing `mail-relay` Service for the sidecar's
  HTTP metrics endpoint.
- Do not create a Service for the sidecar.
- Add a `ServiceMonitor` for the existing `mail-relay` Service's `metrics`
  port.

Maddy remains the only SMTP endpoint visible to workloads.

## Prometheus Metrics

Expose Prometheus metrics on `METRICS_ADDR` at `/metrics`.

Required counters:

```text
pgp_gateway_messages_received_total
pgp_gateway_messages_encrypted_total
pgp_gateway_messages_refused_total{reason="no_key|multiple_keys|policy|malformed|oversize|encryption_error"}
pgp_gateway_messages_relayed_total
pgp_gateway_messages_relay_failed_total
```

Counter semantics:

- Increment `messages_received_total` after a complete DATA payload is accepted
  from Maddy.
- Increment `messages_encrypted_total` only after a key rule matched and
  PGP/MIME encryption succeeded.
- Increment `messages_refused_total{reason="no_key"}` when no key rule matches.
- Increment `messages_refused_total{reason="multiple_keys"}` when configuration
  ambiguity creates more than one match.
- Increment `messages_relayed_total` only after SMTP2GO accepts the encrypted
  message.
- Never attach recipient, sender, subject, or key fingerprint labels. The
  gateway has one job and low cardinality is more useful than per-address
  labels.

## Container Build

Use a multi-stage Docker build:

- Builder: official Go image.
- Runtime: Alpine with CA certificates available for
  SMTP STARTTLS validation.
- Run as non-root.
- Read-only root filesystem.
- No shell dependency at runtime.

The binary should support a startup config validation path. Invalid config,
invalid public key, or fingerprint mismatch should exit non-zero.

## Tests

Required tests:

- Config loading rejects missing values, bad addresses, invalid durations, and
  fingerprint mismatches.
- Key rule matching covers exact addresses, wildcard catchall addresses,
  no-match refusal, and multiple-match refusal.
- Inbound SMTP rejects:
  - multiple recipients
  - recipients with no matching key rule
  - oversized messages
  - malformed messages
- PGP/MIME output has the expected `multipart/encrypted` structure.
- A generated test key can decrypt the encrypted payload.
- Original attachments survive inside the encrypted MIME payload.
- Outbound SMTP uses STARTTLS before AUTH.
- Upstream temporary failures become `4xx` responses.
- Encryption/config/policy failures become `5xx` responses.
- Prometheus counters increment for received, encrypted, refused, relayed, and
  relay-failed messages.

Manual acceptance test:

1. Sync the updated mail relay.
2. Send a test message through Maddy using the existing relay user.
3. Confirm SMTP2GO accepts the message.
4. Confirm the received message is PGP/MIME encrypted.
5. Confirm Proton can decrypt it.
6. Confirm SMTP2GO-visible content does not include the original body or
   original application subject.

## Implementation Steps

1. Create `sidecars/pgp-gateway` Go module and Dockerfile.
2. Implement config loading and public-key fingerprint validation.
3. Implement PGP/MIME construction with tests.
4. Implement wildcard key-rule matching and startup key validation.
5. Implement outbound SMTP2GO client with STARTTLS-required PLAIN auth.
6. Implement localhost SMTP server policy and delivery path.
7. Add Prometheus counters and the HTTP `/metrics` endpoint.
8. Add the GitHub Actions image workflow.
9. Wire the sidecar and ServiceMonitor into `services/platform/mail-relay`.
10. Update restore/readiness docs for the public key, metrics, and SMTP2GO
    ownership.
11. Render and validate the Kubernetes manifests locally.
12. Perform one live mail test through Maddy after review.
